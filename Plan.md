# Plan for Refactoring Pokémon Battle Simulation

**Context:**
The goal is to overhaul the combat mechanics within `src/services/simulator.ts` to provide a robust, multi-turn battle simulation that determines a winner while logging every action taken into a comprehensive, structured JSON object (Battle History). This refactor moves away from placeholder logic and implements core game loop functionality, acknowledging that data persistence will eventually use an external NoSQL database, but using mocked/defined services for the initial implementation.

**Approach:**
The implementation will follow a staged approach to ensure correctness:

1.  **Define State & Types (Typescript):** Create necessary interfaces (`BattleState`, `CombatEvent`, `PokemonStats`) to accurately model the state of the battle and every action within it. These types must be centralized for consistency.
2.  **Implement Core Mechanics:** Create a utility function to handle complex damage calculation, accounting for Type Effectiveness (e.g., Fire > Grass), STAB bonuses, and stat differentials. This encapsulates all combat rules.
3.  **Refactor `simulateBattle`:** Transform the existing placeholder function into a turn-based loop. In each iteration:
    *   Determine the attacking Pokémon/move based on the current state (e.g., alternating turns or random selection).
    *   Calculate damage using the new utility function.
    *   Update both the attacker's and defender's state (HP, status effects).
    *   Create a detailed `CombatEvent` object describing *what happened*, and append it to the battle history array.
    *   Check for victory/defeat conditions and break the loop when necessary.
4.  **Update Logging:** Refactor `getBattleHistory`: This function will no longer mock data; instead, it will return the final structured JSON log generated by the completed `simulateBattle` run.

**Critical Files to be Modified:**
*   `src/services/simulator.ts`: Primary modification site for the simulation loop and history retrieval.
*   `src/types/Team.ts` (or a new file): To house or update core interfaces like `PokemonStats`, `MoveData`, and `CombatEvent`.

**Reused Utilities:**
*   The existing structure of `simulateBattle(team1: Team, team2: Team)` will be maintained as the primary entry point. We will abstract internal logic into helper functions to keep this file clean (e.g., `calculateDamage()`, `updatePokemonState()`).

**Verification Plan:**
After implementation, the following steps must be executed to verify functionality:
1.  **Unit Tests (Recommended):** Write unit tests for the new `calculateDamage` utility function covering edge cases (Type immunity/resistance, status effect mitigation).
2.  **Simulation Run Test:** Call `simulateBattle` with two simple, known-outcome Pokémon pairs and assert that the returned history array contains a sequence of events leading correctly to the declared winner and final HP states.
3.  **History Read Test:** Verify that calling `getBattleHistory(id)` returns the full, structured JSON log generated by a successful simulation run.
